# One Trick Pony - Goldrush Gauntlet 2026 Writeup  
## Description 
Found this log in portal source code, but there seems to be only one superruser that is created with assess to some super secret information. Can you figure out how to get access to it? 
Target: `nc 0.cloud.chals.io 32216`
Attachment: [onetrickpony.tar.gz](onetrickpony.tar.gz)

## Solution
In this challenge we are tasked with somehow finding a superuser account which was located in the source code and use it to get access to the flag.

Extract the file that we were given and we are met with 3 files: 
1. chall.py
2. Dockerfile
3. flag.txt 

When we first load up the chall.py file, we are greeted to our target. 
![Pony Image](ponyImage1.png) 

There is a admin user that is created named alice, but it has a password that is randomly generated by rng.getrandbits(22). When looking futher into the code we see this: 
![Dashboard Image](ponyImage2.png)

This shows that when we correctly login into alice account, we will get the flag. Now knowing what we have to do, lets try connecting to our target: 
![Connect Image](ponyImage3.png) 

When you enter the number 1 for loging, it will then ask you for a username and password. We know that the username will be alice, but for the password, it can have 4,194,304 possible values. If we entered each passsword one at a time, this challenge would basically take forever. Lets try creating an account, and then logging into that account and see what we should get after logging in. 
![Login Image](ponyImage4.png)  

In CTFs the standard solution to this is pipelining. Which is basically sending a bunch of guesses in chunks without waiting for a response. Usually the problem with this approach though is lets say we get the correct password and we move from the login menu to the dashboard menu. The payload will keep making guesses  into the dashboard menu, which will then eventually crash the server, therefore, it can't give us the flag. 
As you can see by the image, the challenge was intentionally setup in a way that the user would send a  1, string, and then an integer. So when we run our pipeline, it would send a 1 for the login, then a string ("alice") for the username, and then a guess of a 22 bit integer. Lets say the guess was correct, the server won't crash because when it still follows the pipeline, it will type 1 for Edit Settings, then types a string ("alice") for new app name, and then types an integer for the OTP bit number, therefore the server is never crashing and we can get our flag.  

### The Exploit 

We created a python script that sends out 4.2 million possibiles of the password and also using the same logic I said above. This python script also has a part where it catches the flag after the successful login. 
The full exploit script can be found here: [pipe.py](pipe.py)
